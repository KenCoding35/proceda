# SOP-Bench Content Flagging: 31% TSR (Benchmark Has Non-Deterministic Tools)

## Summary

We ran Proceda against the SOP-Bench content_flagging benchmark — a 168-task evaluation where
LLM agents must follow a content moderation SOP involving bot detection, trust scoring, severity
assessment, and final decision. Proceda achieves **31.0% TSR** (52/168) with **100% ECR** — every
task ran to completion with all tools called successfully, but the final decisions differ from
ground truth due to **non-deterministic tools in the benchmark**.

| Agent | Model | TSR | ECR |
|-------|-------|-----|-----|
| **Proceda** | **Gemini 2.5 Flash** | **31.0% (52/168)** | **100%** |
| ReAct (best baseline, Table 5) | DeepSeek R1 | 60% | 90% |
| ReAct (2nd best, Table 5) | Claude 3.5 v2 Sonnet | 60% | 90% |
| FC (Claude 3.5 v2, Table 4) | Claude 3.5 Sonnet v2 | 0% | 0% |
| ReAct (Claude 3.5 v2, Table 4) | Claude 3.5 Sonnet v2 | 10% | 90% |

## The Non-Determinism Problem

### Root Cause

The `calculateBotProbabilityIndex` tool contains `bpi = random.random()` — it generates a
random Bot Probability Index on every call, ignoring the input parameters. Similarly,
`calculate_device_consistency()` returns `random.random()`. These random values flow through
the entire pipeline:

```
random BPI (0-1)  ──┐
                    ├──> calculate_user_trust_score ──> determineFinalDecision ──> final_decision
random DCS (0-1)  ──┘
```

The ground truth in the CSV was computed with one set of random values. Each new run produces
different random values → different trust scores → different final decisions. This is a
**benchmark design issue**, not an agent quality issue.

### Impact on Scoring

The randomness explains the prediction pattern:

| Predicted | Should Be | Count | Explanation |
|-----------|-----------|-------|-------------|
| warning | user_banned | 38 | Random BPI much lower than CSV's ~0.95 → lower final score |
| warning | allowed | 24 | Random BPI raises score above 40 threshold for tasks that should pass |
| allowed | removed | 16 | Random BPI near 0 → trust score inflated → low final score |

The random BPI averages ~0.5, but the `user_banned` cases in the CSV had BPI ~0.95. So the
tool returns much lower BPI values → the bot modifier is weaker → the final score drops below
the `user_banned` threshold (>80). This is why **0 out of 53 `user_banned` tasks** were
correctly predicted.

### Why the Baseline Gets 60%

The baseline agents (ReAct with DeepSeek R1 or Claude 3.5 v2) face the same randomness but
achieve 60% TSR. Possible explanations:
1. They ran the benchmark multiple times and reported the best run
2. The random seed was fixed during their evaluation
3. They used a different evaluation methodology

The paper's Table 4 shows **0% TSR for FC and 10% TSR for ReAct** with Claude 3.5 Sonnet v2
on this domain — which is more consistent with our observation. The 60% from Table 5 uses
different models (DeepSeek R1, Claude 3.5 v2) and may have been run under different conditions.

## What Proceda Did Right

Despite the low TSR, the execution was flawless:

- **100% ECR** — Every task completed successfully
- **0 tool failures** — All 4 tools called correctly in all 168 tasks
- **Correct tool ordering** — BPI → Trust Score → Severity → Final Decision
- **Correct parameter threading** — BPI and device consistency from Step 1 correctly passed to Step 2
- **4 missing predictions** — Only 4/168 tasks had extraction issues

The 4-step SKILL.md generated by `proceda convert --tools` worked perfectly:
1. Calculate Bot Probability Index → outputs BPI + device_consistency_score
2. Calculate User Trust Score → uses BPI + device_consistency from Step 1
3. Assess Content Severity → independent calculation
4. Determine Final Decision → combines all metrics

## Technical Changes Made

- **Harness**: Infer `input_columns` from CSV when not in metadata.json (content_flagging
  metadata lacks this field)
- **Output extractor**: Handle bare string values from tools (not wrapped in a dict) when
  there's a single expected column
- **Harness**: Generalized task ID detection to support `content_id`, `video_id`,
  `partner_id`, `order_id` etc.

## Conclusion

Content flagging is an anomalous domain in SOP-Bench. The tools contain `random.random()`
calls that make results non-deterministic. Proceda's structured execution works perfectly
(100% ECR, correct tool chaining) but can't overcome the randomness in the tool
implementations. The 31% TSR reflects random chance alignment between our run's BPI values
and the ground truth's BPI values, not agent quality.

To properly evaluate this domain, the benchmark would need to either:
1. Fix the tools to use deterministic CSV lookups (like all other domains)
2. Seed the random number generator consistently
3. Accept a range of valid outputs per task based on BPI bounds
