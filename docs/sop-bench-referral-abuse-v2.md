# SOP-Bench Referral Abuse Detection v2 ("Hard")

## Summary

We ran Proceda against the SOP-Bench referral_abuse_detection_v2 benchmark — a 200-task
evaluation where LLM agents must detect referral abuse violations using temporal pattern
analysis, historical violation context, and a risk-severity-based enforcement matrix.

Unlike most other domains, the SOP and CSV ground truth are in 100% agreement on this
domain. All failures are **genuine LLM reasoning errors** on the complex multi-step
scoring calculation.

**Best baseline (Table 5):** 98% (Claude 4 Opus FC)

### Key Finding: Thinking Models Perform Worse

Counter-intuitively, Gemini 2.5 Pro (a thinking model) performs **significantly worse**
than Gemini 3 Flash (a non-thinking model) on this domain. The failure rate more than
doubles: 25.5% vs 11.5%. The thinking mode appears to cause the model to over-analyze
and second-guess the straightforward arithmetic scoring, particularly when deciding
whether "No Violation" should win over a violation category that also meets its threshold.

### Gemini 3.1 Pro: Best Quality, Worst Quotas

Gemini 3.1 Pro showed the best quality (9/9 perfect on initial tasks) but is a preview
model with only 250 requests/day on Google AI Studio. At ~16 LLM calls per task, this
allows only ~15 tasks/day — not viable for a 200-task benchmark. It is not yet available
on Vertex AI where quotas would be higher.

## What Makes This Domain Hard

This is the "hard" version of referral abuse detection. Compared to v1:

- **5 violation categories** (vs 4 in v1): adds Temporal Fraud Pattern
- **Weighted indicators** (+2 instead of +1) for strong fraud signals
- **Higher thresholds**: Abusive and Misleading need ≥4 (vs ≥3), No Violation needs ≥6 (vs ≥4)
- **4-tier risk severity**: CRITICAL/HIGH/MEDIUM/LOW based on financial impact and history
- **7 possible enforcement actions** (vs 3 in v1): the action depends on both violation type AND risk severity
- **3x token usage**: 41K tokens/task vs 13K for v1, reflecting more complex reasoning

The scoring step requires the LLM to:
1. Compute 5 scores with weighted indicators across ~30 data fields
2. Determine the highest-scoring category above threshold
3. Calculate risk severity from a separate set of financial/historical criteria
4. Look up the enforcement action in a 2D matrix (violation type × risk level)

## SKILL.md Comparison

Two different SKILL.md versions were used, both generated by `proceda convert --tools --output-fields`:

### 7-step version (converted by Gemini 3 Flash)

Steps 1-5 call the 6 tools. Step 6 calls `determine_enforcement_action`. Step 7 does
all scoring, risk severity, and enforcement lookup in one mega-step. Used with Gemini 3
Flash for the 88.5% run.

### 10-step version (converted by Gemini 3.1 Pro)

Steps 1-5 call tools (reordered: temporal before traffic). Step 6 is OPTIONAL
(documentation). Step 7 computes violation scores. Step 8 calculates risk severity.
Step 9 calls `determine_enforcement_action` and applies the matrix. Step 10 outputs
the final decision with XML tags. Used with Gemini 2.5 Pro and Gemini 3.1 Pro.

The 10-step version decomposes the complex Step 7 into three steps (7, 8, 9), which was
hypothesized to help with reasoning. In practice, the model quality matters more than
the step decomposition.

## Failure Analysis: Gemini 3 Flash (200 tasks)

| Category | Count | % of failures |
|----------|-------|---------------|
| No Violation wins but LLM picks violation | 14 | 60.9% |
| Inconclusive vs No Action confusion | 7 | 30.4% |
| Risk severity miscalculation | 2 | 8.7% |
| **Total failures** | **23** | — |

### Primary Failure Mode: No Violation Not Recognized (14 tasks)

The most common error: No Violation scores ≥6 (the highest category), but the LLM
also notices a violation category meeting its threshold and picks the violation instead.
This is a "highest score" selection error — the LLM needs to compare all scores and
pick the maximum, but tends to fixate on detected violations.

Examples:
- ACC100001: no_viol=6 vs mislead=4 → LLM picks mislead → "Permanent Account Closure" instead of "No Action"
- ACC100056: no_viol=6 vs mislead=4 → same error
- ACC100189: no_viol=6 vs temporal=4 → LLM picks temporal → "Account Closure" instead of "No Action"

### Secondary Failure Mode: Inconclusive Confusion (7 tasks)

4 tasks predicted "No Action" when the correct answer is "Inconclusive" (no score meets
any threshold). 3 tasks predicted "Inconclusive" when the correct answer is "No Action"
(No Violation score ≥6). The LLM confuses the distinction between "no evidence of
violation" (No Violation ≥6 → "No Action") and "insufficient evidence" (nothing meets
threshold → "Inconclusive").

### Risk Severity Errors (2 tasks)

2 tasks correctly identified the violation type but miscalculated the risk severity level,
leading to the wrong enforcement action (e.g., "Account Closure" instead of "Temporary
Suspension").

## Rate Limiting

### Gemini 3.1 Pro (preview model)

- **Quota:** 250 requests/day (Google AI Studio, per-model)
- **Impact:** At ~16 LLM calls per task, only ~15 tasks can run per day
- **Result:** 9 tasks completed (all correct) before quota exhaustion
- **Not available on Vertex AI** (preview-only), so no alternative quota path
- **Workaround:** None viable. Would take ~14 days for 200 tasks.

### Gemini 2.5 Pro

- **Quota:** ~1000 requests/day (Google AI Studio, paid tier)
- **Impact:** At ~16 LLM calls per task, ~55-60 tasks can run per day
- **Result:** 55 tasks completed before quota exhaustion
- **Requires thinking mode** (`budget_tokens >= 128`), which increases latency
- **Workaround:** Use `--resume` to continue in daily batches. Remaining 145 tasks will
  require ~2-3 more days.

### Gemini 3 Flash

- **Quota:** Much higher (ran 200 tasks without hitting limits)
- **No thinking mode required**

## Comparison with v1

| Metric | v1 ("Easy") | v2 ("Hard") |
|--------|-------------|-------------|
| TSR | 95.5% | 88.5% |
| ECR | 100% | 100% |
| Tools | 3 | 6 |
| Violation categories | 4 | 5 |
| Enforcement actions | 3 | 7 |
| CSV bugs | 9 (4.5%) | 0 |
| Reasoning errors | 0 | 23 (11.5%) |
| Avg tokens/task | 13,449 | 41,165 |
| Total cost | $1.64 | $4.59 |

v1 achieved perfect SOP-following (all failures were CSV bugs). v2 reveals genuine
reasoning limits — the multi-step scoring with weighted indicators, threshold comparison,
and 2D enforcement matrix exceeds what Gemini 3 Flash can reliably compute in a single
step.

## Resuming the Gemini 2.5 Pro Run

To continue the partial run (55/200 tasks completed):

```bash
export GEMINI_API_KEY=$(pass soprun/GEMINI_API_KEY)
uv run python -m benchmarks.sop_bench.harness \
  --domain referral_abuse_detection_v2 \
  --max-tasks 200 \
  --resume
```

This will skip the 55 completed tasks and run the remaining 145. Expect ~55 tasks per
daily quota window, requiring 2-3 more runs on subsequent days.
